# CoreShift Architecture

CoreShift is strictly layered. It isolates the pure state machine logic (`core`) from the side-effecting OS interfaces (`low_level`, `runtime`).

## Architecture Map

1. **`core`**
   - **Responsibility**: Pure state machine, reducers, scheduler, replay logic, and invariant validation.
   - **Modules**: `types`, `state`, `engine`, `reducer`, `scheduler`, `lifecycle`.
   - **Rules**: Contains strictly pure business logic. It does not spawn threads, perform IO, or access the OS. State transitions yield `Effect` objects instead of executing them.

2. **`high_level`**
   - **Responsibility**: Addon semantics, Android command mapping, identity management, and capability rules.
   - **Examples**: Maps an abstract `Command` (like `Cmd` or `Dumpsys`) into an execution specification. Manages `PreloadAddon` and `CapabilityToken`.

3. **`mid_level`**
   - **Responsibility**: IPC framing and request/response transport at the process boundary.
   - **Rules**: Translates incoming framed bytes from `low_level::reactor` streams into high-level events.

4. **`low_level`**
   - **Responsibility**: System calls, reactor (epoll/kqueue), process spawning, and IO primitives.
   - **Rules**: **This is the ONLY layer allowed direct OS/platform-facing API access.** (e.g., `libc`, `AsRawFd`, raw file handling). General utility standard library types (`std::collections::*`, `std::vec::*`) are perfectly fine elsewhere.

5. **`runtime`**
   - **Responsibility**: Side effects, structured logging, and orchestrating process execution.
   - **Rules**: Reads `Effect`s generated by `core` and translates them into calls to `low_level`. Binds the reactor, manages the `Tick` loop, and handles actual file routing for logs.

## The Critical Layering Boundary

To ensure the engine remains deterministic, testable in-memory, and portable:
**Higher layers must never reach into raw OS primitives directly.**

- **Bad**: `core::engine` calling `std::fs::read_to_string` or `std::process::Command::new()`.
- **Good**: `core::engine` returning an `Effect::StartProcess` which the `runtime` later evaluates using `low_level::spawn`.

## State Machine Model

Execution relies on a pure loop:
1. `runtime` waits for an `Event` from the `low_level::reactor` (e.g., IPC data or IO completion) or a Time Tick.
2. `runtime` feeds `Event` into the `core::Core::handle_event`.
3. `core` produces pure `Action` transitions (via reducers) and stores new `ExecutionState`.
4. `core` yields `Effect`s.
5. `runtime` consumes `Effect`s and executes them against the OS.
